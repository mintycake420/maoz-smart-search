MAOZ · Smart Search · Part B

A semantic match should be visible, local and falsifiable

This POC turns synthetic Salesforce-shaped profiles into four Hebrew search aspects, ranks them with exact dense retrieval and a transparent concept vocabulary, cites the source span that carried the match, and can decline to return a weak list. A BM25 leg survives as a confidence-gate opener and a diagnostic — an ablation measured its ranking contribution at zero, so it does not rank.

CLAIM

The assignment’s decisive moment is not ten plausible names. It is one testable bridge: a query for חינוך בלתי פורמלי should find evidence about תנועות נוער even though the phrases share no word. Raw FP32 and final quantized BGE-M3 both placed the target second, so the final implementation makes that miss visible and adds a transparent, staff-owned concept bridge. It keeps exact terms in a separate Hebrew-aware lexical leg, exposes which source field won, and records the failed alternatives rather than attributing the final result to the model alone.

02The implemented path

The original query is the only vector computed during search. Seventy-two profile-aspect vectors, source spans and approved concept phrases are prepared once, shipped with an identity manifest, and scanned exactly inside the eligible relation.

01 · Fixture

Validate

Synthetic Salesforce-style Contact JSON only; an explicit marker and synthetic ID prefix are mandatory.

02 · Projection

Split into four aspects

Keep source field and provenance beside each span; restricted fields never enter the aspect contract.

03 · Preparation

Normalise and embed

Conservative Hebrew normalisation; local BGE-M3 creates profile, source and staff-owned concept-phrase vectors in an explicit build step.

04 · Load

Verify artifacts

Check corpus, golden-query, gazetteer, projection, concept-vocabulary and vector-file hashes, plus aspect identity/order, vector shape, finite values, and encoder identity before use.

05 · Query

Scope, bridge, encode locally

Apply caller scope and hard filters, match explicit concept triggers, then encode the original query with the repository-local ONNX model.

06 · Retrieve

Exact dense, ranked alone

NumPy takes the best cosine across the live query and approved concept phrases. Candidate-scoped BM25 still runs over exact terms, clitics and narrow aliases, but only to open the confidence gate on an exact match and to record a diagnostic — it does not rank.

07 · Decide

Aggregate, then abstain

One winning aspect per person, then a synthetic-only dense threshold that may return no result. Weighted reciprocal-rank fusion used to sit here; an ablation across the golden set measured its contribution at zero and it was removed, so gate, tier, score and evidence now all read from a single ranking.

08 · Explain

Return a source span

After rank and gate are fixed, select an original field span, attach provenance, hide raw scores and render without generated summaries.

Visible bridge, no RAG answer

config/concepts.json maps concepts to phrases, never people or profile IDs, and leaves lexical ranking unchanged. There is no instruction-following LLM or generated biography. Evidence is copied from the synthetic source field; its post-ranking selection and its highlight never feed back into rank or the confidence gate. The highlight is the concept phrase that bridged the query, so the explanation on screen names the same mechanism that produced the match.

Beat 2 · Generalisation

Add a person live, then find them

The evaluator invents a person in the UI — role, organisation, experience, interests — and the demo indexes them on the spot: the same fail-closed synthetic validation, the same four-aspect projection, the same local encoder, held in memory only and gone on restart. Their own paraphrase of the need either retrieves that person with a verbatim evidence span, or abstains.

Nothing on this path is pre-computed, which is the point: it answers "does this only work on fixtures you authored?" with a test the evaluator controls. One caveat said out loud: retrieval is by content, not by name — person names are deliberately not embedded, so searching the new person's name abstains. This beat replaced a frozen MiniLM/FP32 model comparison on 2026-07-21; the superseded measurement (FP32 rank 2, MiniLM rank 11 on the flagship) is retained in the decision trail below, as a recorded result rather than a reproducible one — its artifacts were removed along with the feature.

What a held-out measurement changed

Every query in the table above, and every semantic intent in the acceptance set, is configured in config/concepts.json. Passing them showed the system worked on its own configuration. To find out whether that meant anything, eleven Hebrew queries were written against the profiles that have no concept entry, and the dense leg alone was measured against the assembled pipeline.

The retrieval model was not the weak part

The machinery around it was, in three specific ways. The confidence gate had been calibrated on concept-expanded scores and applied to un-expanded ones; expansion lifts a covered query by roughly +0.09..+0.13 and never lifts an uncovered one, so correct answers sitting at dense rank 1 were discarded as לא נמצאה התאמה חזקה. The lexical gate treated any term appearing in a quarter of the corpus as rare — on 72 aspects, nearly every term — so two incidental words let an ordinary sentence override the dense gate, and because a lexical hit also forced the חזקה label, profiles scoring 0.33 were shown to staff as strong matches while the intended profile was filtered out. And the fixed result floor could not prevent padding, because a firing concept lifts every loosely related profile at once.

All three are fixed, each with a regression test, and the calibration script now fails the build if every calibration positive is concept-covered. The pipeline now reaches the dense leg's own ceiling rather than falling below it. It does not beat it: a later ablation across the whole golden set measured the BM25 fusion leg's ranking contribution at zero — it swapped which held-out query it got right rather than adding one — and the leg was removed on Part A.2's own Stage 2 rule. Two of the eleven queries then became calibration positives, so that the threshold has un-expanded examples to fit against; the evaluator's 7/9 covers the nine that remain genuinely held out. The honest reading is that this system's added machinery was subtracting value until it was measured — which is the argument for measuring one's own additions, not an argument that the additions were unnecessary: the concept bridge is still what moves the required flagship example from rank 2 to rank 1.

Final evaluator

With outbound network access blocked: 8/8 acceptance queries placed the intended profile first, 7/7 acceptance out-of-domain queries abstained, and 7/9 held-out queries ranked the intended profile first. Separately and not as evidence, the 9 calibration examples that selected the threshold still behave (4/4 retained, 5/5 abstained). On an Intel i5-9400F, three consecutive five-run passes had warm flagship medians of 0.806, 0.812 and 1.009 seconds, with individual observations between 0.738 and 1.260 seconds. Ad-hoc loops on a loaded machine reached 2.25 seconds, so this is a load-dependent range, not a fixed figure.

The three groups mean different things and are deliberately not added together. Calibration is the weakest: those examples selected the threshold, so reporting them as a result is circular, and an earlier version of this page did exactly that — it counted 12/12 and 6/6 without noting that 4 and 5 of those were fitted examples. Acceptance is independent of the fit, but every semantic intent in it is present in config/concepts.json, so it remains a regression check on this system's own configuration. Only the held-out set — queries written against profiles with no concept entry, and excluded from threshold fitting — speaks to generalization, and it is still nine authored queries over eighteen authored profiles. Genuine evaluation awaits MAOZ-owned query logs and relevance judgments. These ranks and timings describe this frozen corpus, ~603 MB local bundle and CPU only; they do not estimate production relevance, fairness, Hebrew quality or latency elsewhere.

The one measurement that is not circular

Everything above shares an author with config/concepts.json. scripts/evaluate_judged.py does not: 23 Hebrew queries with graded relevance judgments over the 20-profile companion corpus, written without reading the concept vocabulary, split dev (5) / acceptance (13) / abstention (5). The load-bearing figure: all 18 judged primaries sit at un-gated dense rank 1 — 18/18, enforced by the evaluator's exit code — and the concept vocabulary fired on none of the 23 queries. That is the encoder working unaided on queries written by someone who had not seen the answer key — and it doubles as proof the set is genuinely independent, because queries written while reading the vocabulary would have triggered something.

The gated metrics are lower than the ranking: acceptance MRR 0.769, Hit@3 10/13, nDCG@5 0.655, Recall@5 0.577, with 5/5 correct abstentions. On the first (2026-07-20) run they were MRR 0.923 / Hit@3 12/13 — the drop has a single verified cause and it is the finding, not a regression to hide. This harness deliberately applies the runtime corpus's fitted confidence gate to a corpus it was never calibrated on. When the runtime corpus was renamed and resealed on 2026-07-21, its embeddings sharpened and the fitted gate rose from 0.474 to 0.495 — and dev_04, acc_03 and acc_05 (scores 0.477–0.480) joined the original acc_06 (0.389) as correct-at-rank-1 answers the gate abstains on. Nothing was tuned in response: a single absolute threshold does not transfer across corpora, so per-corpus calibration is a v1 requirement, not a refinement. And Recall@5 has a second cause — every multi-relevant query that does answer returns the primary and drops the grade-2 secondary, so the per-intent margin that suppresses padding is also suppressing useful secondary matches. That is a dial MAOZ may want set the other way.

The concept bridge has to re-earn its place

It is the only remaining ranking component, so its value is re-measured on every run rather than asserted once: 8/8 acceptance with the bridge, 6/8 without, held-out equal at 7/9 because the bridge is deliberately not configured for held-out intents. A separate rank_only_effect block isolates dense rank with and without expansion, excluding the gate and the lexical leg — improved on 2, unchanged on 6, regressed on 0 — because comparing gated acceptance in both arms cannot distinguish a rank change from a score lifted over the threshold. earns_its_place gates the exit code, so the justification cannot quietly stop being true.

One caveat, stated before a reviewer finds it: test_hebrew_acronym_community_center sits at rank 11 in both arms. It passes acceptance through the lexical strong-match gate, not through ranking.

05Artifact identity and security boundary

“Offline” means a prepared artifact, not a hidden first-run download. “Secure” means a narrow local demonstration boundary, not production authorization.

What the prepared bundle promises

  • Seventy-two profile-aspect vectors plus source and concept-phrase vectors are precomputed; arbitrary query vectors are produced locally.
  • The 602,549,616-byte model/tokenizer/licence directory loads with offline settings and no hosted fallback. Its 580,393,844-byte ONNX member is distributed as a release asset and must be placed in that directory as a setup step, never fetched at launch. A SHA-256 digest over every file in the directory is verified before the graph loads, so a manually placed copy is checked rather than trusted.
  • The final graph uses weight-only 8-bit asymmetric MatMulNBits blocks of 128 plus int8 embedding tables under ONNX Runtime CPU 1.26.0.
  • The manifest binds vector files to the corpus, golden queries, gazetteer, projection and concept hashes, and records model identity, dimension and local bundle hash.
  • Loader checks cover aspect order/keys, vector shapes, non-finite aspect/source values and concept-phrase identity; query and profile encoder identities must match.

What the local boundary enforces

  • Unmarked records and Contact IDs outside the synthetic namespace are rejected before projection.
  • The server binds to loopback by default, limits request size, applies restrictive browser headers and does not expose raw ranking scores.
  • The frontend inserts API text as text, not HTML. Returned explanations are extractive source spans, not generated prose.
  • No profile or query text is sent to a hosted model API or an instruction-following model.

The identity distinction that prevents a misleading demo claim

BGE-M3 profile vectors for the sealed corpus are preset; everything else is encoded live. The local encoder handles arbitrary Hebrew queries, and since 2026-07-21 it also embeds evaluator-added profiles at the moment they are submitted — the add-a-profile beat runs the same encoder path a query does, with nothing precomputed. Approved concept phrases are preset but fire only through explicit text triggers. The formerly shipped FP32/MiniLM comparison vectors, which existed only for the named flagship query, were removed together with the frozen comparison feature.

06What this POC does not prove

Part A is the production direction; Part B is a deliberately smaller adapter. The seam is preserved, but the operational claims are not borrowed from the architecture document.

Production directionImplemented POC boundary
Salesforce extraction, deltas and deletion Versioned synthetic Salesforce-shaped JSON only. There is no Salesforce connection and no real member data.
PostgreSQL + pgvector In-process float32 NumPy exact scan and in-memory BM25. This is a correctness adapter; it says nothing about PostgreSQL latency or scale.
Salesforce revalidation + SQL/RLS scope The importable API accepts allowed_profile_ids before ranking. The web demo has no identity provider or production authorization.
Local BGE reranker A local experiment moved the frozen flagship target from raw-dense rank 2 to rank 4. The cross-encoder was rejected and is not packaged.
Model-only semantic bridge Both official FP32 and final quantized BGE ranked the required target second. The POC adds a staff-owned concept-to-phrase vocabulary with no people/profile IDs; MAOZ must own and review it before a pilot.
Operational controls No audit ledger, reconciliation, erasure workflow, restore drill, certificate lifecycle or rate-limiting service is claimed.
Real CRM data density Every narrative field is populated on all eighteen fixtures at near-uniform length — Experience__c runs 66–92 characters across every profile, and the four aspects together carry about 348 Hebrew characters per person — 129 tokens concatenated, with the longest single aspect at 59. Partial fill, extreme length variance and stale entries are not modelled, so retrieval here runs on a tidier corpus than a pilot would face. Per-field, per-sector fill rates from the real org are the first measurement I would ask for.
Real relevance and confidence calibration All examples and labels are synthetic. The confidence threshold and the per-intent margin are POC defaults and must be replaced before a real-data pilot. There are no fusion weights left to replace: ranking is dense-only.
Honest limits
  • The synthetic corpus can demonstrate a mechanism but cannot estimate relevance, fairness, coverage or adoption on MAOZ profiles.
  • The recorded MiniLM result was a deliberately selected control, not a broad Hebrew model comparison — and since 2026-07-21 it is a recorded measurement only, no longer re-runnable from shipped artifacts.
  • A profile added live in the demo is scored against a confidence gate that was calibrated on the sealed corpus, and person names are deliberately not embedded — retrieval is by content, so searching an added person's name abstains. Additions live in memory only and vanish on restart.
  • Concept-aware evidence selection and highlighting occur after rank and gate and never feed back into either. Both now draw on config/concepts.json; an earlier build drew the highlight from the golden set, which is why that file is now barred from the result path by a regression test.
  • The 8/8 acceptance result measures this system against its own concept vocabulary. Only the 7/9 held-out result speaks to generalization, and nine author-written queries over eighteen author-written profiles is a smoke test, not an evaluation. Calibration counts are not evidence at all.
  • A result is gated, tiered and displayed on one aspect. An earlier build tiered on the profile's best dense aspect while showing its best fused aspect, which let a weaker span inherit a strong label; the two disagree for 19 profile/query pairs on this corpus.
  • Recovering the FP32 rank with weight-only quantization is one controlled flagship result, not a general equivalence claim.
  • Manifest and loader checks catch known identity and shape drift; they are not a signed software-supply-chain system.
  • Changing --host can expose the unauthenticated demo and therefore leaves the claimed local boundary.
