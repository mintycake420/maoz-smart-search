MAOZ · Smart Search · Part A.3

Only an approved projection becomes searchable

Information security and authorisation are expressed here as two enforceable gates, fifteen testable controls, and explicit reasons to stop — not as a promise that encryption alone makes a people-search system safe.

SECURITY CONTRACT

Salesforce remains the system of record and the authorisation authority. Gate one decides which fields may leave it; gate two decides which rows a particular viewer may rank. Both gates fail closed. The local index is a sensitive, disposable derivative — including its embeddings — and never a second source of permission truth.

01Contract & scope

“Local” reduces one transfer risk. It does not make profile text, queries, vectors, logs or backups non-personal — each remains inside the same trust boundary.

Data minimisation

Restricted fields are absent, not redacted later.

A field-scoped Salesforce identity plus a committed allow-list prevents donor status, staff assessments and personal circumstances from entering the extractor. A widened permission set aborts the run rather than widening the index.

POC
Purpose boundary

Search supports a relevant introduction, not open people browsing.

The result is a small, evidence-backed shortlist. Raw similarity scores, exhaustive counts, bulk export and a vector endpoint are outside the product. The workflow ends in MAOZ's existing introduction process.

v1
Local inference

No member content crosses into a hosted model API.

Every model runs from a locally hosted open-weight checkpoint under the per-repository licences listed in Part A.2. The target design permits two — an embedder and a reranker; the measured POC ships one, because the cross-encoder was rejected on a measured flagship regression and is not packaged. Since MAOZ confirmed the system is Hebrew, translation was removed from the design, so no model in the system generates text at all — every model scores or encodes. Production inference is egress-restricted; model files are pinned and reviewed as software dependencies.

POC
Audience

Staff-only is the safe default, not a permanent entitlement.

Until MAOZ confirms the intended audience, the POC is limited to named staff and synthetic profiles. Member-facing access is a separate launch decision requiring consent, field-visibility and record-visibility semantics.

before member launch
POC boundary. The repository and demonstration use synthetic profiles. A real-data pilot begins only after MAOZ owns the field allow-list, approves the purpose and retention schedule, names the operational owner, and completes the applicable privacy and security assessment.

03Authorisation in one request

Revocation has to beat relevance. The system proves permission before a hidden candidate can affect a score, rank or result count. Timing is mitigated separately and remains observable.

Entry

Authenticate the viewer through Salesforce SSO

The backend validates issuer, audience, signature, expiry and replay protections, then creates a server-side request context. Browser-supplied role claims never become database predicates directly.

Current truth

Resolve current entitlement and visible records in Salesforce

Confirm the user is active and still assigned the search permission. For member-facing search, Salesforce returns the authoritative set or relation of visible profile IDs for this request.

timeout · invalid token · inactive user · permission removed · scope unresolved → DENY

Bind

Bind identity and allowed IDs to a transaction

The application uses a non-bypass database role, sets the verified viewer context transaction-locally, and materialises the allowed IDs as a request-scoped relation. Cross-viewer connection-pool reuse tests must prove that no prior viewer state survives.

Retrieve

Join authorisation inside exact retrieval, then score and rank

Lexical and vector legs operate over the same authorised relation; only then do scoring and any downstream reranking run. A hidden profile cannot become a candidate and cannot be “removed later.” The shipped POC ranks on the dense leg alone — the fusion stage was removed after an ablation measured its contribution at zero, and the cross-encoder reranker was rejected on a measured regression — but the authorisation join sits upstream of all of them, so the boundary argument is unchanged by which scorers are present.

visible_profile_ids ⋈ profile_aspects → exact cosine → confidence gate

Return

Present the minimum useful evidence and record the access

Return a small shortlist using only viewer-visible fields. The security audit records viewer, purpose, result IDs and policy version; general telemetry excludes raw profile text and raw query content.

Not claimed pgvector approximate indexes apply filters after the index scan; iterative scanning retries that post-filter. It is not true pre-filtering. If ANN is introduced later, MAOZ must separately prove recall and authorisation behaviour.
Claimed here The proposed scale uses exact retrieval. The allowed-record relation is joined in the retrieval statement before any score or rank is computed, and the transaction uses a database role that cannot bypass the policy.
04Threat · control · proof register

A control without a proof is an intention. These fifteen entries name the artefact or drill that lets MAOZ verify each promise and the person who owns what remains.

Proportionality is itself a decision, and it belongs in the open. Most of a member profile is not secret — name, role, organisation and sector are substantially public, because these are leaders and the network exists so they can be found. Sizing every control as though the text were confidential would tax a tool whose whole purpose is discovery. So the weight sits deliberately on the two places where sensitivity is real and non-obvious: the fields never extracted at all (donor status, staff assessments, personal circumstances — a staff member's private written assessment reaching the member it describes is relationship-ending), and the sector label, which in a cross-sector Israeli network can proxy for ethnic origin, religious belief or political opinion and is therefore the most regulated field in the schema despite looking like the least. Where a control is expensive and the staff-only risk is genuinely low, it is staged rather than shipped — the enumeration row below is split for exactly that reason: the parts that cost nothing ship in v1, and only the parts that would throttle a discovery tool are staged.

A revoked viewer retains access
v1
Revalidate active status and search permission against Salesforce at request time; no long-lived local entitlement cache; fail closed. Per-record visibility resolution is added before member launch. Proportionality, considered and declined: a short-TTL entitlement cache (30–60 s) would be the cheaper design and is genuinely defensible. 05Privacy lifecycle & governance

The index is temporary; accountability is not. Each stage needs a decision, an owner and evidence that survives beyond the person who built the POC.

01 · DECIDE

Purpose & inventory

Name the use case, audience, legal basis and prohibited uses. Classify every proposed field and likely sensitive inference before it reaches the allow-list.

02 · COLLECT

Minimum projection

Extract under field-level security, record the manifest and policy version, stop on drift, and keep real data out of development fixtures.

03 · USE

Authorised search

Resolve current row visibility before ranking, minimise the result, prevent enumeration, and audit access for a defined review purpose.

04 · CORRECT

Rectify, withdraw, erase

Propagate Salesforce corrections and withdrawals through source text, derived lexical variants, vectors, caches and restored backups; verify against a searchable test record.

05 · RETIRE

Retain & dispose

Set separate periods for profiles, queries, access audits and backups. Securely delete expired derivatives and rehearse restore and incident-response decisions.

Legal touchpoint · not legal advice

Amendment 13 changes the questions MAOZ must answer; it does not answer them from this diagram.

Amendment 13 to Israel's Privacy Protection Law was approved on 5 August 2024 and came into force on 14 August 2025. It expands enforcement and defines “specially sensitive information” to include, among other things, ethnic origin, political opinions, religious beliefs or worldview. In a cross-sector network, sector labels and narrative text may reveal those attributes by inference. Engineering controls provide evidence for an assessment; they are not a finding of compliance.

Decisions for MAOZ and Israeli counsel

Classify before a real-data pilot

  • What is the specific purpose and what notice, consent or other basis covers semantic discovery?
  • Is MAOZ the controller, holder, or both for each database and service relationship?
  • Do the selected fields or likely inferences constitute specially sensitive information?
  • Which 2017 Data Security Regulations tier applies, and what duties follow?
  • Do registration, notification, DPO or security-officer obligations apply at MAOZ's actual scale and role?
  • What response and retention commitments can operations genuinely meet?

06Residual risks, assumptions & launch gates

These are not defects to hide. They are the conditions under which the design is safe enough to test — and the facts that could change the recommendation.

Residual risks that remain

  • Embeddings remain sensitive derivatives. Access controls reduce exposure; they cannot prove reconstruction is impossible.
  • Inference cannot be fully removed. Rate limits, cohort thresholds and tiered results make enumeration harder, not impossible.
  • Self-hosting creates an operator obligation. Someone must own patching, certificates, backups, restore and incident response.
  • Coverage can encode structural bias. Thin profiles can make a community less findable even when the ranking code is neutral.

Working assumptions · pending clarification

  • Audience: named MAOZ staff only for the POC; no member access until expressly approved.
  • Data: Hebrew-language profiles, confirmed by MAOZ, with English appearing inside records as organisation names, job titles and acronyms; the take-home repository and demo contain synthetic data only.
  • Runtime: all production inference is local and does not send profile text to an external model API.
  • Unknowns: real field structure, current consent/notice, record-visibility rules, run expectations and deletion/retention commitments still need MAOZ confirmation.
Real-data staff pilot

Approved purpose and field manifest; applicable legal/security classification; named operations and privacy owners; tested deletion, restore and audit paths.

Member-facing access

Confirmed consent/notice and field visibility; Salesforce-authoritative per-record authorisation proven under revocation; enumeration exercise passed; correction/withdrawal route available to members.

Approximate search later

A separate design review proving filtered recall, permission behaviour and tail latency. Iterative ANN scanning must not be described as true pre-filtering.

Claude Code and OpenAI Codex were used as authoring and engineering assistants: to explore structure, challenge threat models, draft and revise prose, implement the HTML presentations, and review technical claims and code changes. Codex also implemented the synthetic Part B search module and local RTL UI, evaluated local model quantization and a rejected reranker, added authorization and offline regression tests, and ran the frozen-corpus acceptance evaluation.

Correction · 20 July: Codex opened MAOZ’s public network directory while preparing an optional dataset. Browser output exposed public names, roles and one short professional-background passage in the assistant context. This breached the project’s stricter working rule against sending profile text to a hosted AI, despite the source being public. No person-level content was written to the repository or converted into a fixture; the companion corpus uses only abstract field shape and aggregate role mix. All searchable Part B records remain explicitly synthetic. The rule that was breached still stands, and it is reported here rather than quietly corrected because a disclosure that only lists the tools, and not the one time they were used outside the boundary, is not much of a disclosure.

Authority and claim boundaries

The Privacy Protection Authority's Guide to Amendment 13 and its English convenience translation of the 2017 Data Security Regulations are the legal touchpoints for the assessment questions above. Exact classification and duties depend on MAOZ's facts and should be confirmed by Israeli counsel or the organisation's appointed privacy professional.

The embedding-inversion risk is grounded in Morris et al., Text Embeddings Reveal (Almost) As Much As Text (EMNLP 2023). Its strongest exact-recovery result is a constrained best case — evidence to protect embeddings as data, not a universal recovery rate. The pgvector filtering documentation supports the distinction between exact retrieval and post-filtered approximate scanning; Salesforce's user-mode database-operation guidance establishes the platform mechanism whose concrete use must be validated in the MAOZ org.
