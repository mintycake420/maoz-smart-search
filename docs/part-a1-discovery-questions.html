MAOZ · Smart Search · Part A.1

Five questions I would ask before writing any code

Each question below was chosen for one reason: the answer changes what gets built. Not the settings — the structure. Where an answer forks the system, both roads are shown.

NOTE

These are questions for the people who run the network, not for IT. Every one is phrased so that a non-technical answer still resolves a technical decision — that is the test of whether it was worth asking.

01 Data reality

“Show me three member records you'd call complete. For areas of activity, interests and values — is that a picklist, a free-text bio, or notes a staff member wrote after a meeting? Roughly what share of the network has it filled in, and when was it last touched?”

Why first: it decides whether this is a retrieval problem or a data problem. Semantic search over four sparse tags is theatre.

If

Rich free text, widely populated.

A retrieval project. Embed the narrative fields directly and ship search.

If

Four thin tags on a third of records.

An enrichment project. Draft profile summaries from existing meeting notes, run a completion campaign. Search becomes phase two.

02 Trust boundary

“When a member searches, should they see another member's contact details — or only a name and a ‘request an introduction’ button? Are there fields — donor status, staff assessments, personal circumstances — that must never surface to another member?”

Why it matters: it decides whether authorisation is a setting applied at the end, or a constraint that shapes the whole pipeline.

If

Staff-only for version one.

One index, one trust boundary. Materially faster to ship, and the security surface stays small.

If

Member-facing, with restricted fields.

Permissions enforced inside retrieval. Filter candidates before ranking, and embed a redacted projection of each record — an embedding of a private note is still derived from it.

03 Outcome

“What happens in the ten seconds after a good result appears? Does the member email the person directly, or come to you for a warm introduction — and does anything need to be recorded back in Salesforce?”

Why it matters: it decides whether the output is a list of rows or a set of dossiers — two different products.

If

They take it from there.

A search box. Ranked results with contact details, and the work ends at the result.

If

Staff broker the introduction.

A brokering workflow. A short why this match dossier per person and a request-intro action that writes back to Salesforce, keeping the relationship visible to staff.

↳

Either way, the searches themselves become intelligence: repeated unmet demand for, say, water-tech regulators is something the network managers would want to see.

04 Evidence

“Give me the ten searches from last month that were painful or came up empty — real ones, in the words the person actually used.”

Why it matters: this is the only way to know whether the new system is better than the old one. Without it, “the search is good” is an opinion. The grammar of those queries also picks the retrieval strategy.

If

Attribute lookups — “a fintech regulator in the Negev”.

Vector search with metadata filters, where the structured fields carry much of the weight.

If

Need-shaped — “someone who has survived a board revolt”.

Matching on experience narrative. Retrieval over the professional-trajectory aspect rather than the role field — this is precisely where keyword search fails hardest.

↳

And if the painful searches turn out to be mostly “how do I spell their surname”, the honest answer is that they need better lexical search, not AI.

05 Legitimacy

“If a member searched and saw themselves ranked below someone else — or noticed the results skewing toward one sector — how much damage is that? And have members consented to their profiles being used this way?”

Why it matters: a network built on trust has a failure mode a CRM does not. A result can be technically correct and still corrosive.

If

Ranking and representation are sensitive.

Suppress numeric scores, deliberately diversify the top results across sectors, and audit output composition against the composition of the network itself.

If

Consent constrains where data may travel.

Profile text never leaves MAOZ infrastructure — which is what forces a locally hosted embedding model rather than any third-party API.

↳

A retrieval system that quietly favours the most articulate, most central and most represented members would score well on relevance and work against the mission.

Deliberately not asked

Network size and growth rate, and previous attempts at this inside Salesforce. Both matter — but each is an assumption I can state and defend in the architecture rather than spend one of five slots on. Scale is the clearest case: at a few thousand members, exhaustive vector search is genuinely sufficient, and saying so is more useful than asking.

The language mix was not one of them. An earlier draft of this page listed it here, as an assumption safe to carry. That was wrong, and the correction is worth more than the original claim: the question went to MAOZ by email alongside these five, the answer came back המערכת בשפה העברית, and it retired an entire subsystem — a bilingual indexing pipeline with two translation models and an index-time materialisation step. It is the single largest change to the design in Part A.2. A question that deletes a subsystem before it is built is the cheapest thing in this document.
